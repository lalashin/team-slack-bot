# 봇 가동 위치 · DB · 배포 시 고려사항 (정리)

코드 변경 없이 **의사결정·설계 메모**용으로만 정리한 문서다.

---

## 1. 노트북 전원과 봇 동작

**질문:** 지금처럼 노트북에서 `npm run dev` 로만 돌리면, 전원을 끄면 봇을 쓸 수 없나?

**답:** **맞다.** Node 프로세스가 그 기기에서 돌아가야 Slack(소켓 또는 HTTP)과 연결되므로, **PC를 끄거나 슬립으로 네트워크/프로세스가 끊기면** 봇은 동작하지 않는다.

**개선 방향 (개념만):**

- **항상 켜진 환경**에서 실행 (집/사무실 서버, NAS 등).
- **클라우드**에 프로세스를 올려 24시간 실행 (아래 3절).
- Socket Mode든 HTTP든 **“봇 프로세스가 살아 있는 곳”** 이 하나 필요하다.

---

## 2. SQLite를 쓰는 지금 구조

- DB 파일 경로는 `.env`의 `DB_PATH` (기본 `./data/slack-bot.db`).
- **한 프로세스·한 디스크**에 파일이 있어야 하므로, **서버 한 대에 붙어 있는 단일 파일 DB**에는 잘 맞는다.
- **서버리스(요청마다 새 인스턴스)** 환경에서는 같은 파일을 안정적으로 공유하기 어렵고, **디스크가 요청 간에 유지된다는 보장**이 없으면 SQLite 파일을 쓰기 부적합하다.

---

## 3. “Vercel 배포”와 SQLite · Docker

**배경:**  
- **Vercel** 기본 모델은 **서버리스 함수**에 가깝고, 실행 시간·파일 시스템이 **요청 단위로 일시적**인 경우가 많다.  
- **Bolt 봇**은 **항상 떠 있는 프로세스**(Socket Mode) 또는 **항상 받을 수 있는 HTTP 엔드포인트**가 필요하다.  
- **SQLite 파일**은 **같은 인스턴스·같은 디스크에 영구 저장**이 전제인데, 전형적인 Vercel 배포와는 잘 맞지 않는다.  
- **Docker** 자체는 여러 호스팅에서 쓸 수 있지만, “Vercel에 Docker로 올린다”가 **항상 봇+DB에 최적인 것은 아니다** (제품·요금·실행 모델 확인 필요).

정리하면, **“Vercel + 로컬 SQLite 파일 + 24시간 봇”** 조합은 **그대로 두기 어렵다**고 보는 것이 안전하다.

---

## 4. 개선책 방향 (선택지 요약)

아래는 **이 프로젝트를 운영 배포**할 때 자주 쓰는 조합이다. (구현 여부와 무관)

### A. 호스팅 모델을 “항상 켜진 서버”로 맞춘다

- **Railway, Render, Fly.io, DigitalOcean App Platform, AWS ECS/Fargate, VM 한 대** 등  
  - **Node 프로세스**를 컨테이너 또는 프로세스 매니저로 **상시 실행**.
  - **Socket Mode** 또는 **HTTP + 공개 URL** 중 하나 선택.

### B. DB를 “공유 가능한 관리형 DB”로 바꾼다 (권장되는 확장 경로)

- **PostgreSQL** (Neon, Supabase, RDS, AlloyDB 등)  
- 앱은 **연결 문자열**만 두고, **SQLite 파일 의존 제거** (마이그레이션 작업 필요).
- 서버리스/스케일아웃을 염두에 둘 때 특히 유리하다.

### C. SQLite를 유지하되 “디스크가 붙은 한 컨테이너/VM”에만 둔다

- **단일 컨테이너 + 볼륨** 또는 **단일 VM + 디스크**  
- 인스턴스가 **하나**일 때만 안전. 트래픽·가용성 요구가 올라가면 B로 이전하는 편이 일반적이다.

### D. Vercel을 꼭 쓰고 싶다면

- 봇 본체는 **D 옵션에 맞는 다른 서비스**로 두고, Vercel에는 **웹 대시보드만** 두는 식으로 **역할 분리**하는 경우가 많다.  
- 또는 Vercel이 제공하는 **장기 실행·스토리지** 모델이 요구사항과 맞는지 공식 문서로 확인한다.

## 5. 요약 표

| 항목 | 로컬(노트북) | SQLite 파일만 | Vercel 전형적 서버리스 |
|------|----------------|----------------|-------------------------|
| 전원 OFF | 봇 중단 | — | 인스턴스 개념이 다름 |
| Bolt 상시 연결 | 로컬 프로세스 필요 | 동일 | 별도 설계 필요 |
| DB 적합성 | 단일 기기 OK | 단일 서버 OK | 파일 DB 부적합한 경우 많음 |
