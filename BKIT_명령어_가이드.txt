1. /plan vs /pdca plan 차이
================================================================================

1.1 /plan (레거시 명령어)
────────────────────────────────────────────────────────────────────────────
상태: 구식 (deprecated)
기능: 기본적인 Plan 문서 생성만 지원
사용법: /plan {feature}

특징:
- 단순한 기능만 제공
- 더 이상 권장되지 않음
- 호환성 유지용으로만 남아있음

1.2 /pdca plan (현재 권장)
────────────────────────────────────────────────────────────────────────────
상태: 최신 (v1.5.9)
기능: 더 많은 기능 지원
사용법: /pdca plan {feature}

지원 기능:
✓ Plan 문서 생성
✓ Task 자동 생성
✓ PDCA 상태 추적
✓ Executive Summary 필수 생성
✓ 메모리 업데이트
✓ 블로킹 관계 자동 설정

결론
────────────────────────────────────────────────────────────────────────────
❌ /plan 은 사용하지 마세요 (레거시)
✅ /pdca plan 을 사용하세요 (현재 권장)

더 강력한 기능, 자동화, 상태 추적을 모두 지원합니다.

2. PDCA 관련 명령어 (최우선)
================================================================================

PDCA는 Plan → Design → Do → Check(Analyze) → Act(Iterate) → Report의 6단계입니다.

2.1 PDCA 전체 명령어 목록
────────────────────────────────────────────────────────────────────────────

명령어                  설명                        사용법                      단계
────────────────────────────────────────────────────────────────────────────
/pdca plan             Plan 문서 생성              /pdca plan team-slack-bot   Plan (1)
/pdca design           Design 문서 생성            /pdca design team-slack-bot  Design (2)
/pdca do               Do 단계 가이드              /pdca do team-slack-bot      Do (3)
/pdca analyze          Gap 분석 (Check 단계)       /pdca analyze team-slack-bot Check (4)
/pdca iterate          자동 개선 (Act 단계)        /pdca iterate team-slack-bot Act (5)
/pdca report           완료 보고서                 /pdca report team-slack-bot  Report (6)
/pdca archive          완료된 문서 보관            /pdca archive team-slack-bot Archive (7)
/pdca cleanup          보관된 문서 정리            /pdca cleanup                Cleanup
/pdca status           현재 상태 확인              /pdca status                 Status 확인
/pdca next             다음 단계 안내              /pdca next                   단계 안내
/pdca team             팀 모드 시작                /pdca team team-slack-bot    Team Mode

2.2 각 단계별 상세 설명
────────────────────────────────────────────────────────────────────────────

[단계 1] /pdca plan {feature}
─────────────────────────────
기능:
  - Plan 문서 생성 (docs/01-plan/features/{feature}.plan.md)
  - Executive Summary 테이블 자동 생성 (Problem/Solution/Function UX/Core Value)
  - Task #1 자동 생성 ([Plan] {feature})
  - PDCA 상태 파일 업데이트

결과:
  ✓ Plan 문서 완성
  ✓ Task 자동 생성
  ✓ 다음 단계: Design

[단계 2] /pdca design {feature}
──────────────────────────────
기능:
  - Design 문서 생성 (docs/02-design/features/{feature}.design.md)
  - Plan 문서 내용 자동 참조
  - Task #2 자동 생성 ([Design] {feature})
  - Task #1에 블로킹 설정 (Design은 Plan 완료 후)

선행조건:
  ✓ Plan 문서가 존재해야 함

결과:
  ✓ Design 문서 완성
  ✓ 아키텍처 정의
  ✓ 다음 단계: Do

[단계 3] /pdca do {feature}
──────────────────────────
기능:
  - Do 단계 구현 가이드 제공
  - 구현 순서 체크리스트
  - Task #3 자동 생성 ([Do] {feature})

선행조건:
  ✓ Design 문서가 존재해야 함

결과:
  ✓ 구현 시작
  ✓ 코드 작성 가이드
  ✓ 다음 단계: 코드 구현 후 Analyze

[단계 4] /pdca analyze {feature}
────────────────────────────────
기능:
  - Gap Detector Agent 자동 실행
  - Design 문서 vs 구현 코드 비교
  - Match Rate 계산 (0-100%)
  - Gap 리스트 생성
  - Analysis 문서 생성 (docs/03-analysis/{feature}.analysis.md)
  - Task #4 자동 생성 ([Check] {feature})

결과:
  ✓ Match Rate >= 90% → Report 단계로 진행
  ✓ Match Rate < 90%  → Iterate 단계로 진행

[단계 5] /pdca iterate {feature}
────────────────────────────────
기능:
  - PDCA Iterator Agent 자동 실행
  - Match Rate < 90%일 때만 실행
  - Gap에 따라 자동 수정
  - 최대 5회 반복
  - Task #5, #6... 자동 생성 ([Act-1], [Act-2], ...)

실행 조건:
  ✓ Match Rate < 90%

반복 규칙:
  - 최대 반복: 5회
  - 종료 조건: Match Rate >= 90% 또는 최대 반복 도달

결과:
  ✓ Match Rate >= 90% 달성
  ✓ 다음 단계: Report

[단계 6] /pdca report {feature}
──────────────────────────────
기능:
  - Report Generator Agent 자동 실행
  - Plan, Design, 구현 코드, Analysis 통합
  - Executive Summary with Value Delivered
  - 완료 보고서 생성 (docs/04-report/{feature}.report.md)
  - Task #7 자동 생성 ([Report] {feature})
  - Phase → "completed" 상태 변경

선행조건:
  ✓ Match Rate >= 90% (필수)

결과:
  ✓ 최종 보고서 완성
  ✓ 다음 단계: Archive

[단계 7] /pdca archive {feature}
────────────────────────────────
기능:
  - 완료된 문서들을 docs/archive/YYYY-MM/{feature}/ 로 이동
  - Archive Index 자동 생성
  - PDCA 상태에서 feature 제거 (--summary 옵션 사용 시 요약만 남김)

옵션:
  /pdca archive {feature}              (완전 삭제)
  /pdca archive {feature} --summary    (요약 보존)

선행조건:
  ✓ Report 완성 (Phase = "completed")

결과:
  ✓ 문서 아카이빙 완료
  ✓ 프로젝트 정리

2.3 상태 확인 명령어
────────────────────────────────────────────────────────────────────────────

/pdca status
─────────────
기능:
  - 현재 PDCA 상태 출력
  - 활성 feature 확인
  - 현재 phase 확인
  - Match Rate 확인 (Check 단계일 때)
  - Task 상태 확인

출력 예:
  📊 PDCA Status
  ─────────────────────────────
  Feature: user-authentication
  Phase: Check (Gap Analysis)
  Match Rate: 85%
  Iteration: 2/5
  ─────────────────────────────
  [Plan] ✅ → [Design] ✅ → [Do] ✅ → [Check] 🔄 → [Act] ⏳

/pdca next
──────────
기능:
  - 다음 단계 안내
  - 필요한 명령어 제시
  - Phase 진행 상황 확인

사용처:
  - 다음 해야 할 일이 무엇인지 확인하고 싶을 때
  - Phase 진행 순서를 잊었을 때

2.4 팀 모드 명령어 (Agent Teams)
────────────────────────────────────────────────────────────────────────────

/pdca team {feature}
───────────────────
기능:
  - Agent Teams를 활용한 병렬 PDCA 진행
  - CTO-Lead 자동 조율
  - 여러 Agent가 동시에 작업

필수 조건:
  ✓ Agent Teams 활성화 (환경변수 설정)
  ✓ Dynamic 또는 Enterprise 레벨 프로젝트

팀 구성:
  Dynamic  : 3명 (developer, frontend, qa)
  Enterprise: 5명 (architect, developer, qa, reviewer, security)

/pdca team status
─────────────────
기능:
  - 팀 모드 상태 확인
  - 각 팀원의 작업 상태 조회

출력 예:
  📊 PDCA Team Status
  ─────────────────────────────
  Agent Teams: Available ✅
  Display Mode: in-process
  Teammates: 4 / 4 (Enterprise)
  ─────────────────────────────
  Feature: user-auth
    architect: [Design] in progress
    developer: [Do] waiting
    qa: idle
    reviewer: idle

/pdca team cleanup
──────────────────
기능:
  - 팀 모드 종료
  - 리소스 정리
  - 단일 세션 모드로 복귀

3. 프로젝트 레벨 초기화 명령어
================================================================================

3.1 프로젝트 시작 명령어
────────────────────────────────────────────────────────────────────────────

명령어                  설명                        사용법              레벨
────────────────────────────────────────────────────────────────────────────
/starter               정적 웹 프로젝트 시작       /starter init       Starter
/dynamic               동적 웹 프로젝트 시작       /dynamic init       Dynamic
/enterprise            엔터프라이즈 프로젝트       /enterprise init     Enterprise

3.2 프로젝트 레벨 선택 가이드
────────────────────────────────────────────────────────────────────────────

[Starter 레벨]
───────────────
프로젝트 유형: 정적 웹사이트
기술: HTML, CSS, JavaScript (프레임워크 없음)
예시: 포트폴리오, 랜딩페이지, 블로그
특징: 가장 간단함, 빠른 시작

/starter init
  → README.md
  → index.html
  → styles.css
  → script.js

[Dynamic 레벨]
──────────────
프로젝트 유형: 동적 웹 애플리케이션
기술: Node.js + Express + BaaS (bkend.ai)
예시: Todo 앱, SNS, 전자상거래
특징: 인증, 데이터베이스, API 필요

/dynamic init
  → package.json
  → src/app.js
  → src/handlers/
  → .env 설정

[Enterprise 레벨]
─────────────────
프로젝트 유형: 엔터프라이즈급 시스템
기술: Microservices, Kubernetes, Terraform
예시: 대규모 플랫폼, SaaS, 금융 시스템
특징: 복잡한 아키텍처, 높은 확장성

/enterprise init
  → Monorepo 구조
  → Microservices
  → CI/CD 파이프라인
  → Infrastructure as Code

5. Phase 관련 명령어 (개발 단계별)
================================================================================

명령어                      설명                        내용
────────────────────────────────────────────────────────────────────────────
/phase-1-schema             스키마 & 용어 정의         데이터 구조, 엔티티 설계
/phase-2-convention         코딩 규칙 정의             코드 스타일, 컨벤션
/phase-3-mockup             UI/UX 목업                 HTML/CSS 프로토타입
/phase-4-api                API 설계 & 구현            백엔드 API, Zero Script QA
/phase-5-design-system      디자인 시스템              컴포넌트 라이브러리
/phase-6-ui-integration     UI 통합                    프론트엔드 구현
/phase-7-seo-security       SEO & 보안                 최적화, 취약점 제거
/phase-8-review             코드 리뷰                  품질 검증
/phase-9-deployment         배포                       운영 환경 배포

5.1 각 Phase 상세 설명
────────────────────────────────────────────────────────────────────────────

/phase-1-schema
─────────────
목표: 프로젝트의 데이터 구조 정의
산출물: Schema 문서, 엔티티 다이어그램
예시:
  - 사용자 엔티티 정의
  - 관계 매핑
  - 데이터 타입 결정

/phase-2-convention
──────────────────
목표: 팀 전체가 따를 코딩 규칙 정의
산출물: Convention 문서
예시:
  - 파일명 규칙 (camelCase, snake_case)
  - 함수 명명 규칙
  - 디렉토리 구조
  - 주석 규칙

/phase-3-mockup
──────────────
목표: UI/UX 프로토타입 작성
산출물: HTML/CSS 프로토타입
기능:
  - 디자이너 없이 UI 만들 수 있음
  - 최신 UI/UX 트렌드 적용
  - Next.js 컴포넌트로 쉽게 변환

/phase-4-api
────────────
목표: 백엔드 API 설계 및 구현
기능:
  - API 엔드포인트 정의
  - 요청/응답 형식 명세
  - Zero Script QA (로그 기반 테스트)

/phase-5-design-system
──────────────────────
목표: 재사용 가능한 UI 컴포넌트 라이브러리 구축
산출물: 디자인 시스템 문서, 컴포넌트 라이브러리
예시:
  - Button, Input, Modal 등 기본 컴포넌트
  - 디자인 토큰 (색상, 폰트 등)
  - 컴포넌트 문서 (Storybook)

/phase-6-ui-integration
───────────────────────
목표: 프론트엔드 구현 및 API 통합
산출물: 완성된 UI 코드
기능:
  - 상태 관리 (Context API, Redux 등)
  - API 클라이언트 통합
  - 폼 처리 및 검증

/phase-7-seo-security
────────────────────
목표: 검색 최적화 및 보안 강화
기능:
  - Meta 태그 추가
  - Semantic HTML
  - 보안 취약점 제거
  - OWASP Top 10 체크

/phase-8-review
───────────────
목표: 전체 코드 품질 검증
기능:
  - 아키텍처 일관성 확인
  - 컨벤션 준수 여부 확인
  - Design-Implementation Gap 분석
  - 잠재적 문제 감지

/phase-9-deployment
───────────────────
목표: 운영 환경에 배포
기능:
  - CI/CD 파이프라인 설정
  - 환경 설정 (dev, staging, prod)
  - 배포 자동화
  - 모니터링 설정

6. 유틸리티 명령어
================================================================================

명령어                      설명                        사용 시기
────────────────────────────────────────────────────────────────────────────
/code-review                코드 리뷰                   구현 완료 후
/simplify                   코드 정리                   Check >= 90% 후
/zero-script-qa             로그 기반 테스트            API 구현 후
/plan-plus                  고급 Plan (브레인스토밍)    복잡한 기획 필요 시
/memory                     메모리 관리                 기억할 정보 저장 시

6.1 /code-review
────────────────
기능:
  - 코드 품질 분석
  - 버그 감지
  - 모범 사례 확인
  - 개선 제안

사용 시기:
  - PR 전에 코드 검토하고 싶을 때
  - 코드 품질을 높이고 싶을 때

6.2 /simplify
──────────────
기능:
  - 변경된 코드 검토
  - 중복 제거
  - 효율성 개선
  - 문제점 수정

사용 시기:
  - /pdca analyze 결과 Match Rate >= 90% 후
  - 코드를 더 간단하게 하고 싶을 때
  - /pdca report 전에 코드 정리하고 싶을 때

권장 순서:
  /pdca analyze
    ↓ (Match Rate >= 90%)
  /simplify
    ↓
  /pdca report

6.3 /zero-script-qa
────────────────────
기능:
  - 테스트 스크립트 없이 API 검증
  - Docker 로그 모니터링
  - 실시간 JSON 로깅 기반 검증
  - 버그 자동 감지

사용 시기:
  - API 구현 완료 후
  - Phase 4 (API Design & Implementation) 진행 중

6.4 /plan-plus
───────────────
기능:
  - 일반 /pdca plan 보다 고급 기능
  - 브레인스토밍 단계 포함
  - 의도 발견 (Intent Discovery)
  - 대안 탐색 (Alternatives Exploration)

    ↓

  /pdca do team-slack-bot
  → 코드 작성 시작
    ↓

  /pdca analyze team-slack-bot
    ↓

    ✓ >= 90%  → /simplify (선택)    → /pdca report
    ✗ < 90%   → /pdca iterate       → /pdca analyze (반복)
    ↓
[Phase 6] 보고:
  /pdca report team-slack-bot
    ↓
[Phase 7] 정리:
  /pdca archive team-slack-bot
    ↓
완료! ✅

12. 자주 묻는 질문 (FAQ)
================================================================================

Q1. /plan과 /pdca plan의 정확한 차이는?
───────────────────────────────────────
A. /plan은 레거시 명령어로 더 이상 권장되지 않습니다.
   /pdca plan은 최신 버전으로 더 많은 자동화와 기능을 제공합니다:
   - Task 자동 생성
   - PDCA 상태 추적
   - Executive Summary 필수 생성
   → 항상 /pdca plan을 사용하세요!

Q2. 여러 feature를 동시에 진행할 수 있나요?
──────────────────────────────────────
A. 가능합니다. 하지만 권장되지 않습니다.
   - 한 번에 하나의 feature에 집중하세요
   - /pdca team (Agent Teams)을 사용하면 병렬 처리 가능
   - Dynamic/Enterprise 레벨에서만 지원

Q3. Match Rate가 뭐예요?
──────────────────────
A. Design 문서와 구현 코드의 일치도입니다 (0-100%).
   - >= 90% : 보고서 작성 가능
   - < 90%  : /pdca iterate로 개선 필요

Q4. /simplify는 언제 써요?
────────────────────────
A. /pdca analyze 결과 Match Rate >= 90% 후에 사용하세요.
   순서: /pdca analyze → /simplify → /pdca report

Q5. Error가 발생했어요. 어떻게 하죠?
───────────────────────────────
A. 상황에 따라:
   - Plan 문서가 없으면: /pdca plan 실행
   - Design 문서가 없으면: /pdca design 실행
   - 진행 상황이 불명확하면: /pdca status 실행
   - 다음 단계를 모르면: /pdca next 실행

Q6. 언제 /plan-plus를 사용하나요?
─────────────────────────────
A. 요구사항이 복잡하거나 모호할 때:
   - 여러 구현 방식 중 최선을 선택해야 할 때
   - 브레인스토밍이 필요할 때
   - 더 높은 품질의 기획 문서가 필요할 때
   (일반 /pdca plan 대신 사용)

Q7. Agent Teams는 뭐예요?
──────────────────────────
A. 여러 AI Agent가 동시에 PDCA 단계를 병렬로 진행하는 기능입니다.
   - Dynamic: 3명 (developer, frontend, qa)
   - Enterprise: 5명 (architect, developer, qa, reviewer, security)
   - 조건: 환경변수 CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1 설정 필수

Q8. 이전 프로젝트 정보를 기억할 수 있나요?
──────────────────────────────────────
A. 네, /memory 명령어로 중요 정보를 저장할 수 있습니다.
   - 프로젝트 특성
   - 팀 규칙
   - 중요 결정사항
   → 미래 세션에서 자동으로 참고됨
